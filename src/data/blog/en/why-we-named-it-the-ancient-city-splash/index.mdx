Before I adopted dbt and a proper version-controlled transformation workflow, I had a recurring nightmare: a stakeholder asks why a number changed, and I genuinely don't know. A stored procedure was modified. An Excel formula was updated. A BI tool filter was tweaked. The chain of custody is gone.

That problem is solved when business logic lives in Git. Here's what actually changed.

## The Audit Trail Is Real

When every transformation is a SQL file in a repository with commit history, you can answer "what changed and when" in under a minute. `git log --follow -p path/to/model.sql` gives you a full history of every modification, who made it, and the commit message explaining why.

That's not a small thing. In regulated industries, it's often legally required. In every industry, it's the difference between "something changed" and "here's exactly what changed on Thursday at 2pm."

## Reviews Catch Logic Errors Before Production

When a transformation is a pull request, a second pair of eyes sees it before it reaches a dashboard. I've caught more business logic errors in code review than in testing — not because the tests were bad, but because a reviewer asks "why are we filtering out refunds here?" and the answer reveals an assumption that was never documented.

Code review for data models is not overhead. It's the most efficient quality gate in the pipeline.

## Documentation Stays Current Because It Lives Next to the Code

A YAML file in dbt sits next to the model it documents. When you change the model, you change the doc in the same commit. When you don't, the reviewer notices. This is the only documentation system that has ever worked for me — because it's structurally coupled to the thing it describes.

Wikis and Notion pages drift. Schema docs in dbt don't, because they're enforced at the same time as the code.

## Rollback Is Possible

When something breaks, you can revert a dbt model to its last known good state in minutes. Not hours. Not "we need to find who made this change and figure out what it was." Minutes.

That's the difference between a data incident that lasts 20 minutes and one that lasts two days.

## The Onboarding Payoff Is Compounding

Every new analyst or engineer who joins the team gets a repository they can clone, a set of models they can run, and a history they can read. They understand the data because they can read the code that defines it. That's not true when logic lives in a BI tool's hidden calculated field or a stored procedure last touched in 2019.

The upfront investment in a version-controlled transformation layer pays back every time someone new joins the team, every time something breaks, and every time a stakeholder asks why the numbers are what they are.
